# v0.1（建立多 Agent 协作文档）

## 版本号

v0.1

## 版本分配依据

人工未指定版本号。本轮是项目协作体系的首个基线，不开发新玩法功能，因此从 `v0.1` 开始。

## 背景

当前项目已有 SwiftUI iOS 战棋原型、核心玩法规则、测试和 README，但需要建立长期可迭代的 Agent 协作系统。目标是让后续每轮都按“人工目标 -> Agent A 设计提示词 -> Agent B 实现测试 -> Agent C 验收并更新核心逻辑文档 -> 人工复核 -> 下一轮”的流程推进。

## 目标

- 创建标准入口 `AGENT.md`。
- 创建 `update_log.md`。
- 创建 `md/prompt/` 版本化提示词目录，并放入本轮 v0.1 提示词。
- 创建 `md/test/test.md`，写明测试分层、命令、触发条件和当前基线。
- 创建 `md/flow/flow.md`，记录当前真实架构和运行流程。
- 创建 `md/flow/flowchart.md`，用 Mermaid 图展示核心数据流、执行流和 Agent 流程。
- 更新 `README.md`，指向标准协作入口和文档目录。
- 更新 `Tools/verify_project.mjs`，让结构检查覆盖新增核心文档。

## 非目标

- 不新增玩法功能。
- 不修改 SwiftUI 界面。
- 不修改核心规则。
- 不重构 `GameState`、`GameViewModel` 或 `BattleView`。
- 不补 UI 测试。

## 当前架构依据

- 核心规则：`Sources/RomeLegionsCore/GameState.swift`。
- UI 状态：`RomeLegionsApp/App/GameViewModel.swift`。
- 根入口：`RomeLegionsApp/App/RomeLegionsApp.swift` 和 `RomeLegionsApp/Views/RootView.swift`。
- 菜单：`RomeLegionsApp/Views/MainMenuView.swift`。
- 战斗页：`RomeLegionsApp/Views/BattleView.swift`。
- 存档：`RomeLegionsApp/App/SaveStore.swift`。
- 核心测试：`Tests/RomeLegionsCoreTests/GameStateTests.swift`。
- 冒烟工具：`Tools/GameplaySmoke/main.swift`。
- 预览渲染：`Tools/RenderBattlePreview/main.swift`。
- 结构检查：`Tools/verify_project.mjs`。

## 实现步骤

1. 阅读 `README.md`、现有 Codex 规范文档、git 状态、关键源码和测试。
2. 将入口规范统一到大写 `AGENT.md`。
3. 创建 `update_log.md`，记录当前状态和本轮 v0.1。
4. 创建 `md/test/test.md`，按 Probe/Fast、Smoke、Stage Regression、Full 分层。
5. 创建 `md/flow/flow.md`，写当前真实数据流、执行流、状态对象、边界和铁律。
6. 先写好 `flow.md`，再写 `flowchart.md`，每张 Mermaid 图前加中文读图说明。
7. 创建本提示词文件，形成 `md/prompt/` 版本化基线。
8. 更新 `README.md` 的协作规范入口。
9. 更新 `Tools/verify_project.mjs` 的 requiredFiles 和关键 token 检查。
10. 运行结构检查，记录结果。

## 测试要求

本轮是文档和结构检查任务，最低必须运行：

```sh
node Tools/verify_project.mjs
```

如未运行 Swift 测试，最终输出必须说明原因：本轮未改 Swift 代码和核心规则。

## 文档更新要求

- README 必须指向 `AGENT.md`，不再使用小写 `agent.md` 作为入口。
- `update_log.md` 必须记录 v0.1。
- `md/test/test.md` 必须写明文档-only 修改可只跑结构检查但需说明原因。
- `md/flow/flow.md` 与 `flowchart.md` 必须同步表达当前架构。

## 验收标准

- 所有要求的文件和目录存在。
- 文档内容结合本项目真实 SwiftUI/Swift 核心架构，不是空泛模板。
- `AGENT.md` 明确 Agent A/B/C 工作流。
- `md/test/test.md` 有可执行测试命令和触发条件。
- `md/flow/flowchart.md` 至少包含核心数据流、回合执行流、多 Agent 迭代流和测试选择流。
- `README.md` 指向标准入口。
- `node Tools/verify_project.mjs` 通过。

## 风险和禁止项

- 禁止把旧小写入口和新大写入口写成两套互相冲突的规范。
- 禁止声称 Swift 测试通过，除非实际运行并记录结果。
- 禁止修改玩法规则或 UI。
- 禁止把构建缓存或 DerivedData 纳入文档目录。
