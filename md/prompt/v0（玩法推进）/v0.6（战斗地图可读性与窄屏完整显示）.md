# v0.6 / 战斗地图可读性与窄屏完整显示

## 版本号和版本分配依据

- 版本号：v0.6。
- 版本标题：战斗地图可读性与窄屏完整显示。
- 分配依据：`update_log.md` 当前最新正式版本为 v0.5。v0.5 是 Agent X 协作循环文档基线，没有推进战斗体验；本轮目标是推进战斗地图 UI 和窄屏布局，因此版本递增为 v0.6。

## 背景

当前项目已经具备六边形地图、地形、城市、单位、战术姿态、将领、敌军意图、攻击预览和战役状态展示。`GameViewModel` 已提供 UI 所需派生数据：选中位置、可移动格、攻击目标、战斗预览、敌军意图和战役状态。

当前战斗地图的主要问题：

- `BattleView.swift` 中 `HexMetrics.tileWidth = max(44, ...)` 在 390x844 竖屏预览下可能让 12 列地图略超出容器，存在裁切风险。
- 可移动格目前主要是小白点，选中格、可达格、攻击目标之间的视觉层级不够清楚。
- 地形、道路、海域、城市据点和势力控制色已有基础，但整体仍偏占位棋盘，战棋感和战略路线感不足。
- 地图内顶部战术状态条、底部图例和棋盘格需要在竖屏、短横屏、宽屏中同时保持可读，不能遮挡关键格子或被侧栏挤压。
- `Tools/RenderBattlePreview/main.swift` 已能渲染战斗页 PNG，是本轮验收布局的关键入口。

本轮是呈现层优化，不改变玩法核心规则。

## 目标

1. 修正战斗地图在 390x844 竖屏、932x430 短横屏、1024x768 宽屏中的完整显示问题，确保 12x8 初始棋盘完整可见。
2. 优化 `HexMetrics` 或地图布局计算，避免窄屏下固定最小 tile 宽度导致横向裁切。
3. 强化选中格、可移动格、攻击目标的视觉层级：可移动格不能只依赖小点，攻击目标和选中状态必须一眼可辨。
4. 增强地图战棋感：道路、海域、城市据点、势力控制、单位兵牌层级更清晰，但仍使用 SwiftUI 原创形状和系统图标，不引入素材或第三方框架。
5. 保持现有交互语义：点击单位选中、点击可移动格移动、点击攻击目标或敌军红色徽标攻击、点击城市显示城市信息。
6. 保持所有状态来源来自 `GameViewModel` 和 `GameState` 的既有派生数据，不在 SwiftUI 中复制核心规则。
7. 必须通过三尺寸战斗页预览图视觉检查，并记录具体命令和结果。

## 非目标

- 不修改 `Sources/RomeLegionsCore/GameState.swift` 中的移动、攻击、AI、任务、胜负、将领或战术规则。
- 不修改 `GameViewModel` 的玩法语义；本轮原则上只读它提供的派生数据。
- 不新增第三方框架、图片素材、版权素材或复刻原游戏资源。
- 不实现地图缩放、拖拽、相机动画、移动动画、攻击动画或完整美术资产管线。
- 不拆分整个 `BattleView.swift`；可以做局部小结构整理，但不要把本轮变成大规模文件重构。
- 不修改存档结构、Codable 模型、Xcode target 架构或核心测试目标。
- 不创建 PR，不引入 `develop`、`smalldata_test`、`codeb/...` 或其他分支流。

## 当前架构依据

- `RomeLegionsApp/Views/BattleView.swift`
  - `BattleView` 用 `GeometryReader` 判断竖屏、短横屏和普通布局。
  - `WarMapView` 负责地图、地块、城市、单位、攻击目标徽标、敌军意图、战术状态条和图例。
  - `HexTileView` 绘制单个六边形地块及地形、城市、单位、可达、选中、攻击状态。
  - `AttackTargetButton` 和 `AttackTargetRing` 展示可攻击目标。
  - `CityBadgeView` 和 `UnitTokenView` 展示据点和兵牌。
  - `HexMetrics` 计算 tile 尺寸、地图 origin、单位中心点和攻击按钮缩放。
- `RomeLegionsApp/App/GameViewModel.swift`
  - 提供 `focusedPosition`、`reachablePositions`、`attackTargets`、`attackPreview(for:)`、`enemyIntentSummaries`、`primaryEnemyIntent`、`campaignStatus` 等地图展示数据。
  - 负责 `selectTile(_:)`、`attack(_:)` 等命令入口。
- `Tools/RenderBattlePreview/main.swift`
  - 使用 `ImageRenderer` 渲染 `BattleView`。
  - 默认选中 `rome-legion-1`，追加相邻迦太基弓兵，用于复现可攻击目标。
- `md/test/test.md`
  - 战斗页布局变更必须渲染 932x430、390x844、1024x768 三张预览图并检查。
  - 默认完整 SwiftPM、Gameplay Smoke 和 Xcode build 由 `main` push 后的 GitHub Actions 重验证承担。

## 实现步骤

1. 开始前读取本提示词、`AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`、`README.md`、`md/prompt/README.md`。
2. 同步仓库：

   ```sh
   git fetch origin
   git switch main
   git pull --ff-only origin main
   git status --short --branch
   ```

   若没有 `origin`、无法拉取、当前不是 `main` 或存在无法判断归属的无关改动，先报告阻塞，不要继续实现。

3. 阅读相关源码：
   - `RomeLegionsApp/Views/BattleView.swift`
   - `RomeLegionsApp/App/GameViewModel.swift`
   - `Tools/RenderBattlePreview/main.swift`
   - 必要时查看 `Sources/RomeLegionsCore/GameState.swift` 的类型定义，但不要改核心规则。
4. 先确认当前地图尺寸计算：
   - 核对 `HexMetrics` 的 `horizontalUnits`、`verticalUnits`、`widthBased`、`heightBased`、`tileWidth`、`tileHeight`、`totalWidth`、`totalHeight`。
   - 重点修正 `max(44, ...)` 造成窄屏裁切的问题。
   - 新计算必须让完整棋盘在容器内居中，且不让首列/末列、首行/末行被裁掉。
5. 优化 `HexMetrics`：
   - 允许 tile 宽度在窄屏下低于 44，但要保证单位兵牌、城市徽标和攻击徽标仍可读。
   - `actionScale` 应跟随 tile 尺寸合理缩放，避免小屏攻击按钮遮挡单位或战术状态条。
   - 如需要，为地图网格保留顶部/底部/左右安全边距，避免与 `TacticalStatusStripView`、`MiniLegendView` 重叠。
   - 对极小尺寸做防御性处理，避免负 origin、NaN 或布局崩坏。
6. 优化地图视觉层级：
   - `isSelected`：使用清晰的六边形描边、内发光或半透明填充，不能只靠白边。
   - `isReachable`：使用六边形浅色填充、虚线边框、路径感标识或其他明确手段，不能只靠小白点。
   - `isAttackTarget`：保留红色强提醒，并让 `AttackTargetRing` 和 `AttackTargetButton` 在小屏不互相遮挡。
   - 所有状态叠加顺序必须稳定：地形底色 < 地形纹理/路线 < 城市 < 单位 < 意图/攻击提示。
7. 增强战棋感但保持原创：
   - 道路格可增加路线纹理或中线，让战略通道更明显。
   - 水域可增加低调波纹或更清楚的海陆边界。
   - 城市格和城市徽标要更像据点，显示所属势力和城防时不压住单位兵牌。
   - 单位兵牌可以加强军团、骑兵、弓兵、舰队差异，但不要引入图片资产。
8. 保持交互：
   - `WarMapView` 仍通过 `viewModel.selectTile(tile.position)` 和 `viewModel.attack(target.id)` 触发行为。
   - 不要在 SwiftUI 中重新计算移动范围、攻击范围或 AI 意图。
   - 若继续使用 `onTapGesture`，应至少确保可点击地图元素有合理 accessibility label；若改成 `Button`，必须保持视觉样式和点击区域稳定。
9. 必要时更新 `Tools/RenderBattlePreview/main.swift`：
   - 仅当需要更好覆盖地图状态时调整预览场景。
   - 可以保留当前相邻敌军攻击 demo，也可以补充更能暴露可达格、城市、敌军意图的状态。
   - 不要让预览工具写入仓库中的截图产物。
10. 文档更新：
   - `README.md`：如本轮新增或规范三尺寸预览命令，补齐宽屏预览命令。
   - `update_log.md`：追加 v0.6 记录，写清核心变更、关键文件、验证结果和遗留事项。
   - `md/flow/flow.md` / `md/flow/flowchart.md`：只有当真实流程或架构说明变化时才更新；单纯视觉调优不需要硬改流程图。
   - `md/test/test.md`：只有当测试命令、触发条件或预览基线变化时才更新。
   - `Tools/verify_project.mjs`：只有当新增必须检查的结构 token 时才更新，不要为了本轮视觉调优强行扩大结构检查。
11. 不要求修改 `.github/workflows/ci-results.yml`。如果 Agent B 判断必须同步 `CI_VERSION` 到 v0.6，需说明理由，并同步 `md/test/test.md` artifact 示例，同时运行 YAML 解析检查。否则 Agent C 以最新 run 的 `commitSha`、`runId`、`runAttempt` 为准验收，不得因 artifact 名称仍含旧 CI_VERSION 而使用旧 run。
