# v0.4（战役目标与胜负结算）

## 版本号

v0.4

## 版本分配依据

人工未指定版本号。`update_log.md` 最新正式记录为 v0.3，且 v0.3 主要升级 `main` 直推、GitHub Actions 结果包和 Agent C 复判流程，没有推进玩法本身。本轮目标是把现有任务、战斗、占城、招募和 AI 压力收束成可完成、可失败的战役闭环，因此版本递增为 v0.4。

## 背景 / 目前进展

当前项目已经具备以下基础：

- 纯 Swift `RomeLegionsCore` 中已有六边形地图、地形、城市、阵营、资源、军团、移动、攻击、反击、占城、招募、科技、外交、城市扩建、训练、将领、主动技能、战术姿态、AI 回合和敌军意图。
- `GameViewModel` 已负责选择态、命令态、敌军意图摘要、战局态势摘要和 UI 友好的派生数据。
- `BattleView` 已显示地图、资源、战术状态、任务面板、敌情面板、外交、科技、行动按钮和日志。
- `Mission` 当前只有 `title`、`objective`、`reward`、`isCompleted`，`evaluateMissions()` 通过硬编码 id 判断完成并发奖励。
- 当前没有明确的战役胜利、失败、游戏结束、结束后命令保护，也没有能让玩家知道“这局是否已经打完”的核心状态。
- v0.3 已建立云端结果包流程，但 `update_log.md` 中“无 origin”的遗留事项可能已经过期；Agent B 必须以本轮实际 `git remote -v` / `git fetch origin` 结果为准。

本轮选择“战役目标与胜负结算”，因为它能最大化复用现有玩法系统，让原型从功能集合推进为可玩的一局战役。不要把本轮做成大范围 UI 重构或单纯文档整理。

## 目标

1. 在核心层新增明确的战役状态模型，让 `GameState` 能回答当前是进行中、罗马胜利还是罗马失败。
2. 让任务目标从纯展示文本升级为可由核心规则判断的目标要求，减少 `evaluateMissions()` 对 mission id 的硬编码依赖。
3. 保持任务奖励只发放一次；任务完成、战役胜利和战役失败都要产生清晰中文消息。
4. 战役结束后，玩家命令和 AI 推进不得继续改变战局；只读预览、敌军意图查询和已有状态展示可以继续工作。
5. `GameViewModel` 暴露 UI 友好的战役状态、当前目标、结束状态和命令可用性。
6. `BattleView` 最小范围展示战役状态和结束结果，让玩家能看出“正在推进什么 / 已胜利 / 已失败”。
7. 新增或更新核心测试、Gameplay Smoke、结构检查和必要文档。
8. 更新 CI 版本标识为 v0.4，保证 GitHub Actions artifact / manifest 能体现本轮版本。

## 非目标

- 不扩大战役地图、不新增大型关卡编辑器。
- 不实现完整存档列表、继续游戏、删除存档 UI。
- 不做战斗动画、移动动画、相机动画或美术资产替换。
- 不重写 AI 为多回合搜索；只允许为战役状态做必要的 AI 停止保护。
- 不把核心胜负规则写进 SwiftUI `body`、按钮闭包或生命周期回调。
- 不引入第三方框架。
- 不拆掉现有 `MissionPanelView`、敌军意图、战术姿态、将领技能或外交功能。
- 不创建 PR、`develop`、`smalldata_test`、`codeb/...` 或其他长期分支。

## 设计要求

### 核心模型

在 `GameState.swift` 中新增小而清晰的战役状态类型，建议形态如下，可按源码风格调整命名：

- `CampaignStatusKind`：`ongoing`、`romanVictory`、`romanDefeat`。
- `CampaignStatus`：包含 `kind`、`title`、`detail`、`isGameOver`，可选包含 `primaryMissionID` 或 `progressText`。
- `GameState.campaignStatus`：只读计算属性，不要作为必须持久化的状态字段。

胜负规则建议：

- 进行中：默认状态，显示当前未完成的主目标。
- 罗马胜利：所有核心任务目标已经满足。当前任务应至少覆盖“占领叙拉古”“拥有 5 支罗马部队”“占领迦太基”。
- 罗马失败：罗马失去所有城市。不要因为临时没有罗马单位就立刻失败，因为只要还有城市和资源，玩家仍可能招募恢复。

如果 Agent B 认为需要更稳妥的失败规则，可以保持“无罗马城市”为唯一硬失败条件，并把其他压力只作为 UI 文案，不要扩大成本。

### 任务目标

把 `Mission` 从纯文本清单推进为可判断目标，建议新增一个可 Codable 的轻量目标类型，例如：

- `MissionRequirement.controlCity(cityID:faction:)`
- `MissionRequirement.factionUnitCount(faction:atLeast:)`

要求：

- 新增字段必须兼容旧存档。优先使用可选字段，例如 `requirement: MissionRequirement?`；如果加入非可选 Codable 字段，必须实现自定义 `init(from:)` 给旧存档默认值。
- `newCampaign(mode:)` 创建任务时填入 requirement。
- `evaluateMissions()` 优先使用 requirement 判断，旧 id switch 只能作为旧存档或缺 requirement 的兜底。
- 任务奖励只能发一次。已完成任务再次评估不得重复加资源。
- 对完成消息保留中文，例如 `任务完成：夺取西西里。`

### 战役结束保护

新增 `GameRuleError.campaignAlreadyEnded` 或等价错误，中文提示应说明战役已结束。

要求：

- 会改变核心状态的玩家命令在战役结束后不得继续执行，包括移动、攻击、招募、科技、训练、扩建、外交、将领技能、休整、跳过单位和结束回合。
- `performSimpleAI(for:)` 在战役结束后应直接返回空消息或明确“战役已结束”的消息，不得继续招募、移动或攻击。
- `aiIntents(for:limit:)` 是只读预测入口，可以继续返回预测，用于 UI 复盘；不得因战役结束而改状态。
- 一个导致胜利或失败的命令必须先完成本次结算，再让后续命令被保护。例如占领迦太基后应完成占城、任务奖励和胜利消息，然后才进入结束状态。

### ViewModel / UI

`GameViewModel` 只做 UI 派生，不复制胜负规则：

- 暴露 `campaignStatus`、`isCampaignOver`、`campaignStatusTitle` / `campaignStatusDetail` 或类似派生属性。
- `endTurn()` 在玩家回合结束后，如果已经战役结束，不得继续执行 AI while loop。
- AI while loop 每次 `performSimpleAI` 和 `endTurn` 后都要检查战役状态，避免 AI 在已经胜负已定时继续推进。
- 所有命令入口应尊重结束状态；可以通过核心错误统一提示，也可以在 ViewModel 先行禁用，但最终核心必须保护。
- `bannerMessage` 在胜利或失败时应展示战役结果，而不是普通“第 X 回合”消息覆盖。

`BattleView` 最小可见改动：

- 在现有战术状态条或任务面板中展示战役状态，例如“战役进行中 / 罗马胜利 / 罗马失败”。
- 结束时行动按钮、结束回合按钮、招募、科技、外交等入口应禁用或通过核心错误显示结束提示。
- 如果新增独立视图类型，优先放入新文件，例如 `RomeLegionsApp/Views/CampaignStatusPanelView.swift`，不要继续把 `BattleView.swift` 膨胀得更严重；新增文件时必须同步 Xcode project 和本地 typecheck 命令文档。
- 不要在 SwiftUI `body` 中写胜负判断规则。

## 风险和禁止项

- 最高风险是破坏旧存档解码。新增 Codable 字段必须兼容缺字段状态。
- 不要让 `campaignStatus` 依赖 SwiftUI 或 `GameViewModel`。
- 不要让任务奖励在胜利状态反复计算时重复发放。
- 不要让 `aiIntents(for:)` 为预测而修改真实 `GameState`。
- 不要在 `BattleView.body` 中复制“占领叙拉古 / 拥有 5 支部队 / 占领迦太基”等规则。
- 不要因为战役结束保护破坏现有核心测试中的只读预测、战斗预览或状态构造。
- 不要扩大为全量 UI 重构；若需要新增视图，保持小文件、小改动。
- 不要伪造本机测试、云端 run、artifact 或 Agent C 验收结论。
