# v0.25（军团成长优先级读板）

## 版本号

v0.25

## 版本分配依据

`update_log.md` 当前最新功能版本为 v0.24（军团成长决策读板）。Agent X 总目标要求继续向“大征服者罗马”推进，持续优化战斗界面 UI、地图、AI 和将领。v0.24 已让选中单位能看到训练/任命的成本、收益和阻塞，但还缺少全局成长优先级：玩家仍需要逐个点单位判断谁最值得训练、谁最适合任命将领。因此本轮递增为 v0.25，围绕“军团成长优先级读板”继续推进将领与军团成长体验。

## 背景

当前 `GameState.trainingPreview(unitID:)` 与 `generalAppointmentPreview(unitID:)` 已提供单个单位的只读成长决策预览，`GameViewModel.selectedUnitDevelopmentDecisionSummary` 和 `BattleView` 的成长卡展示选中单位训练/任命的成本、预计收益和阻塞原因。

短板是这些信息仍停留在“选中单位”层。更接近策略战棋体验的做法，是在战局层给出成长优先级：哪些军团因受损、临近敌军、缺将领、即将升阶或编制风险更值得训练/任命；哪些行动因资源不足、已任命、非当前阵营或战役结束而暂不可执行。读板必须只读，真实训练和任命仍由既有命令执行。

## 目标

1. 在 `GameState.swift` 新增只读成长推荐报告：
   - `UnitDevelopmentRecommendationKind`
   - `UnitDevelopmentRecommendationPriority`
   - `UnitDevelopmentRecommendationReport`
   - `unitDevelopmentRecommendationReports(for:limit:)`
   - `unitDevelopmentRecommendationReport(unitID:)`
2. 推荐报告至少包含：
   - unitID、kind、priority、score、position
   - cost、canExecute、blockedReason
   - title、summary、detail
   - reasons、impact、projectedRankName、projectedDamageBonus
   - optional candidateName、candidateTrait
3. 核心推荐必须复用 v0.24 的 `trainingPreview` 与 `generalAppointmentPreview`，不得复制成本、候选将领、军阶或收益公式。
4. 推荐评分应解释清楚，至少考虑：
   - 是否可执行。
   - 是否缺将领。
   - 当前生命损失与训练恢复收益。
   - 预计军阶/伤害提升。
   - 附近敌军和军团编制战备情况。
   - 已行动、资源不足、已任命等阻塞。
5. 在 `GameViewModel.swift` 新增 UI 摘要：
   - `UnitDevelopmentRecommendationSummary`
   - `unitDevelopmentRecommendationSummaries`
   - `primaryUnitDevelopmentRecommendationSummary`
   - 选中单位成长读板可继续使用 v0.24 summary，但战局层必须能展示全局首要成长推荐。
6. 在 `BattleView.swift` 新增“成长优先级”展示：
   - 顶部态势 chip 显示首要成长推荐。
   - 完整战局面板展示成长行或小卡。
   - 选中单位情报中的成长卡可显示“全局推荐/本单位推荐”提示，或保持选中单位读板但新增全局推荐卡。
   - SwiftUI 只能展示 ViewModel 派生结果，不得重新计算推荐分。
7. `Tools/RenderBattlePreview/main.swift` 增加云端断言：
   - `primaryUnitDevelopmentRecommendationSummary` 存在。
   - `unitDevelopmentRecommendationSummaries` 非空。
   - 推荐 title、summary、detail、priority、reason/impact、accessibilityLabel 非空。
   - 至少存在一个训练推荐和一个任命推荐摘要。
   - 失败抛出 `missingUnitDevelopmentRecommendationSummary`。
8. Swift Testing 与 Gameplay Smoke 增加核心只读断言：
   - 推荐报告不改变 `GameState`。
   - 训练推荐复用 `trainingPreview` 成本、预计经验/生命/军阶。
   - 任命推荐复用 `generalAppointmentPreview` 候选名、trait 和成本。
   - 缺将领单位应产生任命推荐；已有将领单位的任命推荐应显示阻塞或低优先。
   - 资源不足时推荐仍可读但不可执行，阻塞原因可读。
9. 更新 `.github/workflows/ci-results.yml` 的 `CI_VERSION` 为 `v0.25`。
10. 更新 README、flow、flowchart、test、prompt README、update_log，并把 v0.24 云端通过证据写入 `update_log.md`。

## 非目标

- 不新增升级树、装备、兵种转职、自动训练、自动任命、将领池管理、将领改名或技能选择。
- 不改变训练/任命成本、恢复、经验收益、候选顺序、trait 映射或真实结算。
- 不改变 AI 行动、战斗结算、移动、攻击、技能、城市、外交、任务或胜负规则。
- 不新增第三方框架，不拆分大型 SwiftUI 文件，不做 BattleView 全局重构。
- 不运行任何本地测试、build、typecheck、RenderBattlePreview、`Tools/verify_project.mjs`、`git diff --check`、YAML/JSON/Plist 解析或脚本语法检查。

## 当前架构依据

- `TrainingPreview` 与 `GeneralAppointmentPreview` 已是训练/任命单体决策的唯一核心预览来源。
- `trainUnit(id:)` 与 `appointGeneral(unitID:)` 已复用对应预览写回，推荐报告必须建立在这些预览之上。
- `LegionFormationReport` 已能描述职责、战备、近敌、建议姿态和命令建议，可作为推荐理由来源。
- `WarMeritStatus` 已封装经验、军阶、伤害加成和进度，推荐报告不得在 UI 层复制军阶公式。
- `GameViewModel` 已有战局摘要、军团编制摘要、成长决策摘要、资源短缺文案和无障碍文案转换范式。
- `BattleView` 顶部 chip、完整战局面板、紧凑/完整选中单位情报是本轮 UI 落点。

## 实现步骤

1. 在 `GameState.swift` 的成长预览类型附近新增推荐 kind、priority 和 report 类型。
2. 实现 `unitDevelopmentRecommendationReports(for:limit:)`：
   - 遍历指定阵营单位。
   - 对每个单位尝试生成训练推荐和任命推荐。
   - 复用 `trainingPreview`、`generalAppointmentPreview` 和 `legionFormationReport`。
   - 按 priority、score、unitID、kind 稳定排序并截断 limit。
3. 实现 `unitDevelopmentRecommendationReport(unitID:)`：
   - 返回该单位最高优先级推荐。
   - 缺单位时抛 `missingEntity`。
4. 推荐评分建议：
   - 可执行推荐基础分高于不可执行推荐。
   - 训练优先考虑受损、低经验、临敌、编制 critical/strained、训练后军阶或伤害提升。
   - 任命优先考虑无将领、临敌、攻击/机动单位、编制风险和候选 trait 影响。
   - 不可执行报告仍输出阻塞原因和资源差距交给 ViewModel 解释。
5. 在 `GameViewModel.swift` 增加推荐 summary，把核心 report 转成 title、badge、priorityLabel、kindLabel、unitLabel、costLabel、reasonLabel、impactLabel、statusLabel、accessibilityLabel。
6. 在 `BattleView.swift` 加入成长优先级 chip 和战局/情报展示，沿用现有卡片密度和颜色，不把 UI 扩成独立页面。
7. 更新 `Tools/GameplaySmoke/main.swift`、`Tests/RomeLegionsCoreTests/GameStateTests.swift` 和 `Tools/RenderBattlePreview/main.swift`。
8. 更新 workflow 版本和文档。
9. 完成后只做只读 `git diff` / `git status` 检查，提交并 push `origin/main`，等待 GitHub Actions 和 Agent C artifact 复判。
