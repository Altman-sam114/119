# v0.52 地图主导战斗壳层 UI 重构

## 版本号

v0.52

## 版本分配依据

`update_log.md` 当前最新正式版本为 v0.51（HUD 信号胶囊 UI 重构），本轮沿玩法推进序列递增为 v0.52。v0.50/v0.51 只完成局部重复组件收敛，没有改变战斗页信息架构；人工总目标要求继续向公开截图所体现的“大征服者罗马”式地图主导战棋体验推进，并优先重构战斗 UI，因此本轮进入第一轮可独立验收的战斗壳层重排。

## 背景

公开商店截图可提炼出可借鉴、但不涉及复制素材或品牌的界面规律：顶部资源带很薄，地图占据绝大多数首屏，单位与城市的关键信息直接贴在地图标记上，常用工具只占边缘少量空间，选中单位后出现紧凑命令坞，而不是让常驻重侧栏持续压缩地图。

当前 `BattleView` 与上述层级存在明显差距：

- 非竖屏布局长期保留约 270...340pt、约占视口 30% 的右侧命令栏；短横屏同样固定显示紧凑侧栏。
- 手机竖屏地图高度约为视口的 45%，下方命令 deck 占据首屏剩余空间。
- `CommandPanelView` 顺序堆叠情报、战场、敌情、战局、军令、科技、外交、任务和战报，所有信息同时争夺注意力。
- `UnitTokenView` 已有阵营、兵种、将领星标、姿态、经验、冷却和生命条，`CityBadgeView` 已有城市名与城防；但地图标记仍可进一步强化“关键信息就地读取”，无需新增素材。
- `Tools/RenderBattlePreview/main.swift` 的紧凑命令区与城市读板像素检查仍假设内容位于横屏右侧或竖屏下半部。壳层改成底部命令坞和按需边缘面板后，这些区域必须同步更新，不能通过放松阈值逃避布局验收。

本轮采用原创的罗马军令桌视觉，不下载、不提交、不描摹原游戏截图、图标、人物、地图纹理、商标、文案或其他受保护素材。公开截图只用于判断信息密度和交互层级。

## 本轮目标

1. 把 `BattleView` 改成地图主导壳层：薄顶部带 + 主地图 + 选择驱动的底部紧凑命令坞 + 少量边缘工具入口；默认不再常驻完整或紧凑右侧栏。
2. 保留当前全部读板、命令入口和业务行为，把低频信息放入可关闭、按类别切换的按需战场抽屉，而不是删除或隐藏到不可达状态。
3. 让选中军团、城市或地块的最高频信息和操作在底部命令坞内直接可见；更完整的情报、成长、将领、敌情、内政和战报仍可通过抽屉查看。
4. 在不新增图片资产的前提下强化地图单位/城市就地读板，保持六边格、路线、热区、目标线、反制、技能、敌军意图和现有 HUD 的语义与点击链路不变。
5. 同步 RenderBattlePreview 像素验收、CI artifact 版本和真实架构文档，使云端可以证明三种尺寸下新壳层非空、地图占主导、命令坞可读、城市入口可读且没有明显重叠。

## 当前架构依据

- `BattleView` 当前用 `GeometryReader` 判断手机竖屏、短横屏和普通宽屏，并直接组合 `TopBarView`、`WarMapView`、`PhoneCommandDeckView`、`CompactCommandPanelView`、`CommandPanelView`。
- `WarMapView` 已负责地图、单位/城市兵牌、攻击按钮、敌军路线、军议路线、机动、目标线、反制、热区、控区、技能范围、地图侦察 HUD、敌情闭环 HUD 和图例；本轮不得另建第二套地图或重新计算 overlay。
- `SelectionPanelView` / `CompactSelectionPanelView`、`ActionsPanelView` / `CompactActionsPanelView`、`BattlefieldFocusPanelView`、`EnemyIntentPanelView`、`StrategicBalancePanelView`、`TechnologyPanelView`、`DiplomacyPanelView`、`MissionPanelView`、`LogPanelView` 已承载所有既有信息和命令，可重组到按需抽屉，不能删除。
- `GameViewModel` 继续是选择态、命令态和 UI 派生数据来源；SwiftUI 只展示并调用既有命令方法。
- `Tools/RenderBattlePreview/main.swift` 在渲染前已完整断言所有 ViewModel 读板同源关系，并输出 932x430、390x844、1024x768 三尺寸城市/单位共 6 张 PNG；本轮主要增加/修正壳层像素验收，不应重写既有语义断言。
- `.github/workflows/ci-results.yml` 云端编译时显式包含 `BattleView.swift`，不依赖 Xcode 工程资源列表；本轮禁止触碰用户已有改动的 `RomeLegionsApp.xcodeproj/project.pbxproj` 和 `RomeLegionsApp.xcodeproj/project.xcworkspace/xcuserdata/**`。

## 设计与交互要求

### 1. 地图主导壳层

- `BattleView` 默认状态不再并排显示 `CommandPanelView` 或 `CompactCommandPanelView`。
- 顶部状态带在普通 Dynamic Type 下目标高度约 44...50pt，必须保留菜单、模式/回合/战役状态、banner、五类罗马资源和结束回合；可用 `ViewThatFits`、横向滚动或紧凑图标层级适配，但不得静默丢掉粮草、科学或威望。
- 抽屉关闭时，932x430 与 1024x768 的地图区域应覆盖顶部带以下绝大多数可用面积；390x844 竖屏不再固定把地图限制为 45%，无选择时地图应明显超过首屏高度的一半，出现命令坞时也应保持地图是第一视觉主体。
- 使用稳定的 `frame`、`safeAreaInset`、`ViewThatFits` 或容器约束保持布局，不得使用 `UIScreen.main.bounds`。普通内容不得因为按钮 label 或动态文案变化导致地图尺寸跳动。

### 2. 边缘工具与按需抽屉

- 新增轻量的战场边缘工具入口，建议分类为“情报/军令”“战场”“敌情”“内政/元老院”“战报”，实际命名可按现有中文语义收敛。
- 工具按钮必须使用 SF Symbols + 可访问文本标签，视觉上可以 `.labelStyle(.iconOnly)`；每个交互命中区域至少 44x44，当前选中分类不能只靠颜色区分。
- 点击工具入口后在横屏/宽屏从 trailing 覆盖显示可关闭抽屉，在竖屏可从底部或适合的覆盖层显示；抽屉覆盖地图但不永久挤压地图。关闭后回到完整地图。
- 抽屉必须复用现有 panel 组合并保持所有入口可达：
  - 情报/军令：选择情报、完整军令、将领、成长、姿态、攻击、技能、训练、任命、休整、跳过、城市扩建和招募。
  - 战场：战场焦点、目标线、态势交汇、热区/控区、将领协同、机动和军议等既有读板。
  - 敌情：敌军意图、路线、计划时间线、敌将威胁、反制建议与指令。
  - 内政/元老院：科技、外交、战役任务与推进线。
  - 战报：现有事件日志。
- 不得把展开分类写进 `GameState` 或存档。视图本地状态应为 `private @State`，关闭/切换抽屉不触发玩法命令。

### 3. 选择驱动的紧凑命令坞

- 选中军团、城市或地块后，在地图下方使用稳定高度的紧凑命令坞，不恢复常驻重侧栏。无有效选择时可隐藏或缩成极薄状态行。
- 军团命令坞至少直接显示：单位身份、阵营/兵种、生命、当前姿态、将领/无将领状态，以及当前最高频入口（当前可攻击目标、主动技能、姿态/军令、休整、跳过、更多情报）。不得改变现有 action 或 disabled 条件。
- 城市命令坞至少直接显示：城市名、归属、产出/城防/部署的紧凑摘要，以及扩建、招募和更多城市情报入口；真实成本、部署位置和阻塞原因继续读取 `SelectedCityBrief` / 现有预览。
- 地块或敌方选择至少提供紧凑身份/地形/目标状态和打开完整情报的入口，不要显示一块空白命令区。
- 命令坞可把低频细节交给抽屉，但不能让原有训练、任命、招募、外交、科技等入口从任何尺寸中消失。
- 高频按钮使用图标或图标+短文本，保持至少 44pt 点击区域、明确 disabled 样式、VoiceOver label 和现有无障碍文案。不要新增大段“如何使用”说明文字。

### 4. 地图就地读板

- 保留现有 `HexTileView`、`UnitTokenView`、`CityBadgeView` 及全部 overlay 调用链，不复制核心数值计算。
- 在现有原创代码绘制兵牌上，优先让单位生命/状态、兵种、将领存在与必要的短名称更容易就地扫读；城市继续直接展示城市名与城防。信息必须随 `tileScale` 收敛，窄屏不允许长文字覆盖相邻六边格。
- 阵营、可达、攻击、技能、热区、控区等状态不能只靠颜色；继续保留图标、描边、纹理或徽标差异以及 tile accessibility。
- 地图侦察 HUD、敌情闭环 HUD、战术状态 strip 和主动图例仍须存在。允许为了避免命令坞/抽屉遮挡而重排边距或压缩成现有 compact 形态，但不得删除字段、重新评分或拦截地图点击。

### 5. SwiftUI 质量边界

- 新壳层要提取为职责清楚的专用 `View` 类型，避免继续把所有分支内联进 `BattleView.body`。由于本轮明确禁止修改 Xcode project，新增壳层小类型先放在 `BattleView.swift`，不要创建必须写入 `project.pbxproj` 的新 Swift 文件。
- 不在 `body`、按钮闭包、`onAppear` 或生命周期回调中加入规则计算；按钮只调用现有 `GameViewModel` 命令或修改本地抽屉状态。
- 避免 `AnyView` 和昂贵的 inline filter/sort；保持地图 `ForEach` 的稳定 identity。
- 尊重 Dynamic Type、VoiceOver、Differentiate Without Color 和 Reduce Motion；本轮若不新增动画，不要为了“高级感”加入无意义大动画。
- 延续项目现有暗色军令桌、红/金/青等多层语义色，不复制公开截图的具体素材、构图细节或品牌元素。

## 风险与禁止项

- 风险：底部命令坞可能与 WarMap 底部侦察 HUD、敌情闭环和图例重叠。优先让命令坞参与外层布局并为地图提供稳定可用高度，不要用多个任意 offset 互相避让。
- 风险：把旧侧栏改成抽屉后，某些低频按钮可能失去入口。实现前后按现有 panel 清单逐项核对，不得以“UI 更简洁”为由删功能。
- 风险：RenderBattlePreview 的旧右侧采样区会误判。必须基于新底部命令坞和边缘工具改区域，同时增加地图主体检查，不能简单放宽阈值。
- 风险：顶部过薄可能损害 Dynamic Type 或丢资源。普通字号追求薄，高辅助字号允许合理增高或滚动，不能硬裁切。
- 风险：地图就地标签过多会遮住相邻格。只显示决策必要的短信息，并随 `tileScale` 收敛。
- 禁止复制、裁切、描摹、提交或生成与《大征服者罗马》相同的截图素材、人物肖像、地图纹理、图标、商标或界面文案。
- 禁止把核心规则写进 SwiftUI，禁止为迁就布局修改 AI/战斗/任务/城市结算，禁止让预览与真实命令分叉。
- 禁止删除或弱化既有 RenderBattlePreview 语义断言，禁止伪造 PNG、manifest、JUnit、日志、run 或 artifact。
- 禁止本地运行任何验证命令，禁止把未运行的本地测试写成通过。
