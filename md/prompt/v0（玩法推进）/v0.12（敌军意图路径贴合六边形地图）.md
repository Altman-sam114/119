# v0.12 / 敌军意图路径贴合六边形地图

## 版本号和版本分配依据

- 版本号：v0.12。
- 版本标题：敌军意图路径贴合六边形地图。
- 分配依据：`update_log.md` 当前最新正式版本为 v0.11（将领详情与战术指挥可读化）。本轮继续响应 Agent X 的“大征服者罗马”长期目标，聚焦地图与 AI 意图可读性，因此在 v0.11 后递增为 v0.12。

## 背景

v0.10 已经把敌军意图显示为地图路线、目的地和目标格叠层，但当前路线段仍是 `origin -> destination -> target` 的直线。六边形地图里敌军实际移动会受邻接、地形、单位占用和机动值影响，直线可视化在水域、丘陵、城市和绕行场景下不够像“战术路线”。本轮目标是在不改变 AI 决策的前提下，让敌军意图叠层更贴合六边形格路径。

## 目标

1. 在 `GameViewModel` 为敌军意图新增只读路径派生：
   - 从 `AIIntent.originPosition` 到 `AIIntent.destination` 生成相邻六边形步进路径。
   - 路径段应按相邻格逐段绘制，不再只画单条长线。
   - 若无法找到合法路径，保留当前直线兜底，不能让意图叠层消失。
2. 路径计算应尽量复用现有地图规则：
   - `Position.neighbors(width:height:)`
   - `TerrainType.movementCost`
   - `UnitKind.canEnter(_:)`
   - `GameState.effectiveMovement(for:)`
   - 当前单位占用判断。
3. 路径预览必须只读，不修改 `GameState`、不改变 `performSimpleAI`、不改变 `aiIntents(for:)`、不改变 AI 评分和目标选择。
4. `BattleView` 继续只消费 `EnemyIntentMapOverlay.routeSegments` 并绘制线段；如果需要增强箭头或线段视觉，只限小范围 UI 调整。
5. `Tools/RenderBattlePreview` 加断言：移动后攻击场景的移动路线应包含多个相邻 route segment，且每段 `from` / `to` 必须是六边形邻居；保留 v0.11 的将领详情、姿态预览和紧凑命令区渲染断言。
6. 同步 README、flow、flowchart、test、update_log 和 CI artifact 版本到 v0.12。

## 非目标

- 不修改 `Sources/RomeLegionsCore/GameState.swift` 的 AI 决策、评分、真实移动、战斗结算、战术姿态、将领技能、存档字段或 Codable。
- 不引入寻路第三方库。
- 不把路线预览结果反向写入 `AIIntent` 或 `GameState`。
- 不新增动画、移动执行路径、回放、真正的多步 AI。
- 不创建新分支、PR、`develop`、`smalldata_test` 或候选分支流程。

## 当前架构依据

- `GameState.aiIntents(for:limit:)` 只读预测敌军倾向，返回 `AIIntent.destination`、目标单位/城市和预计伤害。
- `EnemyIntentSummary.routeSegments` 当前在 `GameViewModel.swift` 中由 `originPosition`、`destinationPosition` 和 `targetPosition` 拼出直线段。
- `BattleView.EnemyIntentRouteLayerView` 只遍历 `overlay.routeSegments` 画线，不应知道路径算法。
- `GameState.reachablePositions(for:)` 的私有实现已体现可复用原则：从单位位置出发，按邻居、地形进入能力、占用和移动成本累积可达格。
- `Tools/RenderBattlePreview/main.swift` 已构造确定性移动后攻击场景，适合验证敌军意图路径不再只是单条长线。

3. 在 `GameViewModel.swift` 中调整敌军意图 route 派生方式。建议做法：
   - 让 `EnemyIntentMapOverlay` 存储或接收 `[EnemyIntentRouteSegment]`，不要让 `EnemyIntentSummary.routeSegments` 永远只能返回直线。
   - 在 `GameViewModel.enemyIntentMapOverlays(for:)` 里为每个 summary 调用一个私有 helper，例如 `enemyIntentRouteSegments(for:)`。
   - helper 对移动段先尝试生成六边形路径；对目标段继续保留 `destination -> target` 的虚线 target leg。
   - 路径段 id 使用稳定顺序，例如 `summary.id-route-0`、`summary.id-route-1`。
4. 路径算法建议：
   - 对 `origin == destination` 返回空移动段。
   - 使用 Dijkstra 或 BFS + 成本排序，优先低移动成本，其次短 hex distance，其次稳定坐标排序。
   - 只允许单位可进入地形。
   - 当前被其他单位占用的格不可作为中间格或目的格；但起点单位本身所在格允许作为起点。
   - 总成本应不超过 `GameState.effectiveMovement(for: planningUnit)`；planningUnit 可复制 summary.unit，并应用 intent 的 `tacticalOrder`，但不要改变 state。
   - 若找不到路径，使用当前直线兜底。
5. 若算法放在 `GameViewModel`，必须保持 UI 派生边界：它可以读取 `state.tile(at:)`、`state.unit(at:)`、`state.effectiveMovement(for:)`，但不得调用 mutating 方法或改变 `state.units`。
6. `BattleView.swift` 原则上不需要大改；如需微调可只增强路线段视觉，例如多段箭头在每段末端更清晰。
7. 更新 `Tools/RenderBattlePreview/main.swift`：
   - 保留 v0.10 敌军意图、v0.11 将领/姿态/紧凑命令区断言。
   - 对 `advanceOverlay.routeSegments` 增加断言：至少存在一个非 targetLeg 的移动段，移动段数量在确定性场景中应大于 1，且每个移动段的 `from.neighbors(width:height:)` 包含 `to`。
   - 继续生成三尺寸 PNG，不提交 PNG。
8. 如本轮不改核心规则，`Tests/RomeLegionsCoreTests/GameStateTests.swift` 可不新增用例；如果发现必须暴露核心路径 API，则必须新增 Swift Testing 和 Gameplay Smoke 断言，并说明为什么 UI helper 不足。
9. 更新文档和 CI 版本：
   - `.github/workflows/ci-results.yml` 的 `CI_VERSION` 更新为 v0.12。
   - `README.md` 已实现能力补充敌军意图六边形路径路线。
   - `md/flow/flow.md` 同步敌军意图路径派生数据流。

## 验收标准

- 敌军移动后攻击意图的移动路线由多个相邻六边形段组成，而不是单条长线。
- 目标段仍可用虚线显示 `destination -> target`。
- 找不到路径时仍有直线兜底，地图叠层不消失。
- AI 决策、评分、真实移动、战斗结算和存档字段没有变化。
- 三尺寸预览通过，短横屏/竖屏命令入口仍可见。
- GitHub Actions 最新 v0.12 run 通过，Agent C artifact 验收通过。

## 风险和禁止项

- 禁止把路径预览写入 `AIIntent` 或 `GameState`。
- 禁止让路径 helper 影响 `performSimpleAI`。
- 禁止为了路线美观忽略地形进入能力或单位占用。
- 禁止提交 DerivedData、PNG、xcresult、本地缓存或历史 artifact。
- 如果路径算法导致预览工具不稳定，优先收窄确定性场景和断言，不要放宽到“只要有线就行”。
