# v0.50 战斗读板标签行 UI 重构

## 版本号

v0.50

## 版本分配依据

`update_log.md` 当前最新正式功能版本为 v0.49（AI 作战计划时间线读板），本轮沿玩法推进序列递增为 v0.50。人工最新总目标要求继续向“大征服者罗马”推进，并在新模型下重构既有代码，尤其是战斗界面 UI；本轮选择低风险、可云端验证的 UI 结构重构作为第一步。

## 背景

`RomeLegionsApp/Views/BattleView.swift` 已超过 7600 行，战斗页累计了大量读板、卡片和 HUD。当前多个读板使用几乎相同的“图标 + 标题 + 值 + Spacer”短标签行结构，例如 `BattlefieldConvergenceLabelRow`、`SelectedUnitSituationLabelRow` 和 `SelectedUnitOrderWindowLabelRow`。这些重复结构增加后续 UI 迭代成本，也让战斗界面的代码更难维护。

## 目标

- 在 `BattleView.swift` 内抽取一个通用短标签行组件，例如 `ReadoutLabelRow`，统一处理 system image、标题宽度、文字透明度、lineLimit 和 minimumScaleFactor。
- 将 `BattlefieldConvergenceLabelRow`、`SelectedUnitSituationLabelRow`、`SelectedUnitOrderWindowLabelRow` 的重复实现迁移到通用组件，保留原有调用语义或用轻量 wrapper 兼容现有调用点。
- 保持战斗页视觉、布局高度、字号、颜色透明度和无障碍文案的行为基本不变；本轮是 UI 代码重构，不新增玩法信息。
- 同步必要文档和 CI artifact 版本到 v0.50，让云端结果包明确覆盖本轮。

## 非目标

- 不修改 `GameState`、AI 评分、作战计划、敌军意图、移动、攻击、技能、姿态、任务、城市、训练、任命、外交、存档或胜负结算。
- 不新增新的战斗读板、按钮、自动命令、命令队列、自动反制、自动技能或自动移动。
- 不拆分新 Swift 文件，不修改 `RomeLegionsApp.xcodeproj`，避免本轮把 UI 重构扩大成工程结构变更。
- 不重做整体视觉主题、地图渲染、布局断点、资产系统、装备、升级树、将领池 UI、外交 UI、存档 UI 或建筑树。
- 不引入第三方框架。

## 当前架构依据

- `BattleView.swift` 是 SwiftUI 战斗页主要文件，现有读板组件都在同一文件中定义。
- `GameViewModel` 负责 UI 派生数据；本轮不需要修改 `GameViewModel`。
- `Tools/RenderBattlePreview/main.swift` 已覆盖关键战斗页读板和地图叠层断言，包括 v0.49 的 AI 作战计划时间线读板。
- `.github/workflows/ci-results.yml` 负责云端结构检查、SwiftPM、Gameplay Smoke、RenderBattlePreview 和 Xcode build，并上传结果包。

## 实现步骤

1. Agent B 开始前同步最新 `origin/main`，确认当前分支是 `main` 且工作区没有无关改动。

## 风险和禁止项

- 风险：统一组件若默认参数不精确，可能导致窄屏标签文字缩放或截断变化；必须保留各 wrapper 的标题宽度和 minimumScaleFactor。
- 风险：纯 UI 重构容易被误扩成视觉重做；本轮只处理短标签行复用。
- 禁止修改核心规则、AI 行为、ViewModel 派生字段或 RenderBattlePreview 断言语义来迁就 UI 重构。
- 禁止本地运行测试或构建，也不得把未运行的本地命令写成验证结果。
